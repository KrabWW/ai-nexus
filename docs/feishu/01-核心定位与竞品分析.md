# 01-核心定位与竞品分析

> 来源：飞书知识库 | 最后编辑：2026-04-06

## 核心定位

**AI Nexus = 业务知识治理层**

不是普通数据库，不是通用记忆系统，不是文档库。是为企业研发团队提供的、让 AI 真正理解业务规则的中间层。

## 解决的问题

关键问题不在于"AI 能不能写代码"，而在于：**AI 如何持续理解你的业务世界。**

场景举例：
- AI 修改排班逻辑但不知道 ICU 需要 24h 值班医生 → 生产事故
- AI 重构代码但不知道某个字段在合规审计中是必填 → 违规
- AI 新增功能但不知道它与现有业务规则的冲突 → 返工

## 竞品分析与定位澄清

### OpenViking / mem0 — 通用 Agent 记忆管理

解决「AI 不忘事」：背景文档存储、通用语义检索、会话记忆管理。这是字节的团队在做的事，你打不过，也不需要打。

### AI Nexus — 业务知识治理层

解决「AI 懂业务」：结构化图谱、开发 Hook、知识审核。这是 OpenViking/mem0 完全空白的领域。

### 本质差异

| 维度 | OpenViking/mem0 | AI Nexus |
|------|----------------|----------|
| 数据类型 | 非结构化文档、对话历史 | 结构化实体、关系、规则 |
| 查询方式 | 语义相似度 | 图遍历 + 规则匹配 |
| 知识来源 | 自动摄入 | AI 抽取 + 人工审核 |
| 核心动作 | 检索上下文 | 注入业务约束 |

### 共存策略

两者是**上下游关系**，不是竞争关系。在 MCP Server 中同时暴露 AI Nexus 自有工具 + OpenViking 代理工具，Claude Code 通过同一个 MCP 入口调用。

## 核心创新：业务图谱

业务不是文本。业务是：**实体 + 关系 + 规则**。

```
医生 ── 属于 ──▶ 科室
 │
 │ 排班
 ▼
门诊

ICU ── 需要 ──▶ 24小时值班医生（硬性规则）
 │
 │ 预约
 ▼
患者
```

这就是 AI 最喜欢的数据结构——结构化、可查询、可推理。

## 一句话总结

AI Nexus 让 AI 像资深员工一样懂业务规则。OpenViking 让 AI 不忘事。**两者互补，各司其职。**

## 新增功能：与 mem0 / OpenViking 怎么相处

核心原则是：**你代理它，不是替代它。** AI Nexus 的 MCP Server 里暴露一个 `get_session_ctx` 工具，底层直接调 mem0 的接口。Claude Code 不知道背后是两个系统，它只看到一个 MCP 入口。

这样做有两个好处：一是你的销售故事变成"完整的 AI 开发记忆体系"而不是"又一个记忆工具"；二是你永远不需要做会话记忆功能，mem0 更新迭代你自动跟着受益。

唯一需要注意的边界：mem0 注入的是"上次这个项目讨论了什么"，AI Nexus 注入的是"这个业务领域有哪些硬性规则"。两者都进 context，但优先级不同——业务规则是约束，不可违反；会话历史是参考，可以被新指令覆盖。这个优先级要在 MCP 工具的 description 里写清楚，让 LLM 知道谁的权重更重。

## 和向量库怎么相处

不要运维独立的向量数据库。用 pgvector 挂在你的 PostgreSQL 旁边，这是最轻的方式。

实际使用场景是两类查询分路由：当 Claude Code 问"有没有关于 ICU 的规则"，这是模糊查询，走向量召回；当它问"ICU 的完整约束链是什么，包括所有关联实体"，这是结构化遍历，走图查询。两条路在 `query_rules` 工具内部自动路由，调用方感知不到差异。

从 llm-wiki 文章里学到的一点也适用在这里：Karpathy 说在中等规模（约 100 个来源、几百个页面）下，直接靠结构化索引就够了，不需要向量数据库。你的业务规则图谱在早期大概率也是这个量级——几百个规则节点，图遍历完全够用，向量检索是后期规模化以后的事，不要过早引入复杂度。

## 从 llm-wiki 学到的三件事怎么用

**第一件：用"编译"管道解决冷启动。** 你团队的飞书文档不是直接能进 AI Nexus 图谱的，它们是非结构化的原料。用 llm-wiki 的 ingest 流程做第一步加工——让 LLM 把飞书页面编译成结构化 Markdown，标注实体、关系、规则——然后 AI Nexus 再从这个半结构化的中间层里抽取图谱节点。这比直接从原始文档里抽取准确得多。

**第二件：把 Lint 做成产品功能。** llm-wiki 的 Lint 是检查知识库健康，你的版本更有价值——定期扫描图谱，找互相冲突的规则、找没有任何 PR 曾经触发过的规则节点（可能是死规则或未覆盖的风险）、找最近两周 PR 里违反了但没被系统捕获的案例。这个功能可以做成周报形式推给团队负责人，是很强的留存钩子。

**第三件：PR 捕获事件回写图谱。** 这是你比 llm-wiki 强得多的地方。每次 AI 代码审查捕获到一个业务规则冲突，这个事件本身——冲突位置、触发规则、修正方式——自动回写进图谱，加强对应规则节点的置信度，或者生成新的规则节点。llm-wiki 是单向的（你喂它，它长大），AI Nexus 可以是双向的（规则执行反过来喂养规则库）。这个数据飞轮是你最强的长期护城河。
