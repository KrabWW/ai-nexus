# 02-数据库设计（修订版）

> 来源：飞书知识库 | 最后编辑：2026-04-06

## 设计原则

只做图谱核心三张表：实体、关系、规则。不做文档存储（OpenViking 负责）、不做 UI 资源管理（后期再做）。

## SQLite 表结构

### 1. 业务实体表 (entities)

```sql
CREATE TABLE entities (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    type TEXT NOT NULL,          -- 人物/地点/机构/概念/系统
    description TEXT,
    attributes TEXT,             -- JSON: {"职称":"主任医师","工号":"D001"}
    domain TEXT NOT NULL,        -- 业务领域: 排班/预约/手术/结算
    status TEXT DEFAULT 'approved',  -- pending/approved/rejected
    source TEXT DEFAULT 'manual',    -- manual/ai_extracted/imported
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 2. 关系表 (relations)

```sql
CREATE TABLE relations (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_entity_id INTEGER NOT NULL,
    relation_type TEXT NOT NULL,     -- 属于/排班/需要/预约/依赖
    target_entity_id INTEGER NOT NULL,
    description TEXT,
    conditions TEXT,                 -- JSON: {"时间段":"夜班","优先级":"高"}
    weight REAL DEFAULT 1.0,         -- 关系权重 0-1
    status TEXT DEFAULT 'approved',
    source TEXT DEFAULT 'manual',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (source_entity_id) REFERENCES entities(id),
    FOREIGN KEY (target_entity_id) REFERENCES entities(id)
);
```

### 3. 业务规则表 (rules)

```sql
CREATE TABLE rules (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    description TEXT NOT NULL,
    domain TEXT NOT NULL,            -- 排班/预约/手术/结算
    severity TEXT DEFAULT 'warning', -- error/warning/info
    conditions TEXT,                 -- JSON: {"科室":"ICU","时段":"全天"}
    related_entity_ids TEXT,         -- JSON: [1,3,5] 关联实体ID
    status TEXT DEFAULT 'pending',   -- pending/approved/rejected
    source TEXT DEFAULT 'ai_extracted', -- manual/ai_extracted/imported
    confidence REAL DEFAULT 0.0,     -- AI 提取置信度 0-1
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 4. 知识变更日志 (knowledge_audit_log)

```sql
CREATE TABLE knowledge_audit_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    table_name TEXT NOT NULL,        -- entities/relations/rules
    record_id INTEGER NOT NULL,
    action TEXT NOT NULL,            -- create/update/delete/approve/reject
    old_value TEXT,                  -- JSON snapshot
    new_value TEXT,                  -- JSON snapshot
    reviewer TEXT,                   -- 审核人
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 索引设计

```sql
CREATE INDEX idx_entities_type ON entities(type);
CREATE INDEX idx_entities_domain ON entities(domain);
CREATE INDEX idx_entities_name ON entities(name);
CREATE INDEX idx_relations_source ON relations(source_entity_id);
CREATE INDEX idx_relations_target ON relations(target_entity_id);
CREATE INDEX idx_relations_type ON relations(relation_type);
CREATE INDEX idx_rules_domain ON rules(domain);
CREATE INDEX idx_rules_status ON rules(status);
CREATE INDEX idx_rules_severity ON rules(severity);
```

## 向量存储

AI Nexus **不自建向量数据库**。语义检索是 mem0/OpenViking 的职责（L2 知识治理层共存）。

- **结构化查询**（图遍历、规则匹配）→ SQLite entities/relations/rules 表
- **语义模糊查询**（"有没有关于 ICU 的规则"）→ 代理调用 mem0，mem0 内部使用向量数据库
- 两条查询路由在 `query_rules` 工具内部自动切换，调用方无感知

MVP 阶段（几百条规则）纯 SQLite 图遍历够用，不需要向量召回。规模增长后再接入 mem0 的语义检索。
